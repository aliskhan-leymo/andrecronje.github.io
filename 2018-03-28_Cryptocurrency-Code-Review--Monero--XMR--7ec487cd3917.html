Cryptocurrency Code Review: Monero (XMR)

Disclaimer: These reviews are done as is from what is on display in the master branch of the repo’s made available. This review is not a comment on the overall project, scope, or success thereof. This was done as an educational review by me and any comments in the article are simply my opinion. It should not be used as any comment or advice on the project as a whole.

“Monero: the secure, private, untraceable cryptocurrency”

A lot of work has gone into this.

Calculable values are pre-calculated. This is a good sign of optimization.

Not always.

“Catch all” error statements should be a last resort, first you deal with the ones you expect, and then you catch the rest.

In some places that is the pattern however. Multiple developers tend to implement different patterns.

Perfect pattern, deal with the case you expect, throw the rest, 10/10

Not much else to say, the rest of the code follows similar standards, the tx pool, block, and core are all good, I really wish there was more to point out, but it’s just good. This review feels a bit empty though, but I don’t see value in just pasting more and more examples.

So I guess, it is just good.

So instead, I’m going to talk about the life cycle of code. Code starts out perfect. You get that first scope, and you begin building for it, and it’s exact, it’s neat, it’s everything you want it to be. Then the first change begins to happen, someone decides actually, they want feature x to instead do y. Now, you could go back and build it from the start, which is the right way, but you have timelines to meet and people have expectations, so instead you do a small little hack, then another, then another, and eventually you end up with this

Now I will comment, you can see this in the code, the hardfork splits, and all the calculations that happen around them. But this will inevitably happen as a project ages, there is nothing we can do about it, but how it is dealt with is the difference. So normally as a developer you reach a point where you stop caring about your code and you just keep hacking. But you could still architecture these forks in a clean way, build them around a config and set that config up as a guide through the rest of the code, this is how Ethereum has done it with DAO and Byzantium and this is how Monero is doing it.

It shows that even though the project has aged and even though it has been through hardship, it is still kept to a certain standard. The way these scope creep features are dealt with, is still in a good fashion.

Sure, the newer guys will have the advantage because they get to start clean, optimized, and perfect, but they have yet to walk this road, and trust me, they will, it’s inevitable. But it’s how we deal with these issues in code that shows the caliber, and I think both Ethereum and Monero are good examples of this standard.

Conclusion: It’s optimized, well documented, high quality code.
