This was done as an educational review by me and any comments in the article are simply my opinion. It should not be used as any comment or advice on the project as a whole.

EDIT 08/04/0218

Received my HPB balance, so let’s give this another go. Test 1, highest block was 631 transactions followed by 394, and 356 within the same second window. So around 1381 transactions in a 1 second block(s) window.

Second test, highest block 905 in 1 second block(s) window. Followed by 490, 335 in a 1 second block(s) window.

Concurrent test provided similar results. There were interesting empty blocks in between. I was streaming constantly so the empty blocks confused me a bit, I think perhaps one of the validators wasn’t receiving my transactions when it was it’s sealing turn, I should have actually logged the signatures to check.

Conclusion: So a bit slower than my initial isolated tests, and slower than I was expecting for a PoA hardware optimized implementation. Still 1–1.3k TPS concurrently isn’t bad at all.

HPB have launched their HPB Lightnode Public Testnet. This follows shortly after my code review I did a few days back https://medium.com/@suchi.blackwing/cryptocurrency-code-review-high-performance-blockchain-hpb-2dc856c307e9

At that time I wasn’t overly enthusiastic with their changes, since it didn’t have that much substance and I don’t have the skillset to test the hardware side of things. So I feel this testnet test might also be premature, since I can’t test it in conjunction with the hardware. That being said, let’s see what we have to work with.

Now let’s remember, we are essentially using the baseline of go-ethereum running in PoA, this was from our previous review in which we discovered their Prometheus consensus is just PoA and that their testnet is based on go-ethereum.

Build process is the same as geth, nothing new here.

Testnet genesis, this is a bit strange, why are we doing a genesis? Technically we should just boot up and join the network. But this is an opportunity to give ourselves an alloc, so let’s do it while we are at it.

Added ourselves in.

Genesis set up.

The random number mechanism is the new block sealing mechanism we discussed in the last review, I originally thought this was a per block randomization, but now it seems to be a global state, still, let’s set it.

Ok, so we aren’t syncing, let’s wipe it all and start again from scratch

No DB to wipe, we didn’t actually mine anything, so makes sense, reset and reconfigured.

Still nothing.

Can’t mine, because we aren’t authorized (PoA, so makes sense)

mmm, can submit the transaction, but nothing in txpool.

Ok, let’s try a completely fresh server, everything from scratch (only just now noticed genesis.json is gensis.json)

Completely fresh install, followed to the letter, no connectivity, but again from everything we know about geth this makes sense, since we are creating a genesis file (we are establishing our own private blockchain) and then we are running our own networkid. These are all the steps for running your own private node. So let’s ignore the instructions and do it the old fashioned way.

Still nothing.

Conclusion: Perhaps an error in translation or perhaps the testnet joining instructions aren’t complete yet, but right now I can’t manage to join the testnet, I can run my own private chain just fine (although I can’t mine because prometheus needs at least 3 authorizors [same as Clique])

For now, all I can say, this is still just geth running in PoA, did anyone else manage to get this working?
