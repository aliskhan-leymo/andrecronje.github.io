Cryptocurrency Code Review: RChain

Disclaimer: These reviews are done as is from what is on display in the master branch of the repo’s made available. This review is not a comment on the overall project, scope, or success thereof. This was done as an educational review by me and any comments in the article are simply my opinion. It should not be used as any comment or advice on the project as a whole.

Very active, very busy, very detailed. This team is off to a grade A start.

So the two things I’m excited about;

  1. RHO-VM; these are independent smart contract virtual machines, the implication here is twofold, domain namespace (which isn’t implicit here, but has to be considered) and encapsulation. When we look at Ethereum a big concern is that everything is happening in the same state, if you are transferring EOS or TRX tokens, you are doing them on the primary Ethereum EVM, there is only one. Does an EOS transfer need to be aware of a TRX transfer? While in the future the answer to this question might be yes, currently it is a strong no, so the idea that you can encapsulate each one in it’s own ecosystem is a fantastic and required idea.
  2. Namespaces; again something I have long thought is a concern in Ethereum, this is an extension of point 1 above but also extends to the blockchain as well as the VM, the implication here is self standing eco-systems, while this would require an first class citizen transfer mechanism to fully function, it is required to create ecosystem based scaling.

To know what we are looking at, we have to quickly see how they mange their layout, we see they have 4 essential core items, Rholang, Roscala, Rosettee, RSpace. We are interested in Rosette, RSpace, and I’m assuming node (repository).

I always like to start with the communications layer as well, so let’s jump into comm/

We start off with a Java repo, I was under the impression this will be C++, but that doesn’t change anything. ProtoBuff, Kademlia, all that’s missing is GRPC and then we have the standard 3 pillars of blockchain communication. At this point I think we might as well standardize that communication is ProtoBuff wrapped in GRPC using Kademlia for discover.

Not Java, I stand corrected, we are looking at a scala repo. Scala is a highly functional version of Java (oversimplified I know, don’t attack me). We can see they are using a Monad design pattern (common to see in scala), I’m always glad when I see strong design patterns.

Polite nod to Ethereum, so far these guys are a class act.

This is interesting to see, normally you would just delegate to an existing library, these guys are unpacking the frames themselves. I’m in two minds about this, on the one hand, let other people that specialize in this development focus on it, on the other hand, if you really want to optimize every last drip out of something, you have to custom implement it yourself. Not sure which one of those two these guys are, but let’s continue and see.

Great comments, very detailed and clean exceptions of what still needs to be done, good to see. In Kademlia we also see a very low level custom implementation, same comment as above about letting someone else focus on the library, but again, it looks like these guys are trying to drip every last ounce of performance out of this system.

Fantastic error handling.

When they use code they point it out clearly, great to see.

Tiny spelling mistake, we will forgive them for that. Ok, so fantastic comm/ code, off to a great start.

We head over to the Node code after finishing off the comm/ code. Little bit disappointed here, from the overview it doesn’t look like we will be seeing the namespace blocks just yet, right now it seems that only the REPL and/or P2P are running. So the sexy that we came looking for doesn’t look like it exists yet. But let’s keep going.

So unfortunately, Node didn’t have much substance. HTTP server and P2P listener essentially.

We move onto Rspace

Looks like it is a key-value store, there are a lot of great existing options out there, what’s wrong with using Rocks or LevelDB? We could argue optimization again, but at what point are these guys spreading themselves to thin?

Not much to see in it unfortunately, so let’s move on to Rosette

Rosette is an already existing solution? After having gone through it, I can’t really ascertain what was originally designed and what RChain contributed. I feel a bit wanting for more, so let’s look at Rholang.

Not a lot to point out unfortunately, it is good code though.

Conclusion: So, I’m torn with this one, first off, the code that has been written is high quality code. Very high quality. But it’s missing everything I was excited about. Right now, it’s a P2P module and a Rholang implementation. Where is the blockchain? I’m not trying to discount the amount of work that has gone into this, but at the same time it feels lackluster.

A lot of the code reads like a research experiment, the kind you see in tech labs at universities with a bunch of guys doing their doctorates. It’s really good, but do we need another language? Do we need another p2p implementation? Do we need another key-value store? Could namespaces and encapsulated VM’s not have been incorporate into Ethereum? (The answer by the way is, yes, they can)

Fantastic code, well developed, very far away from product.
