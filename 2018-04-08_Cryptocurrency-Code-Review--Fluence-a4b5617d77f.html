Cryptocurrency Code Review: Fluence

Disclaimer: These reviews are done as is from what is on display in the master branch of the repo’s made available. This review is not a comment on the overall project, scope, or success thereof. This was done as an educational review by me and any comments in the article are simply my opinion. It should not be used as any comment or advice on the project as a whole.

Fluence is a decentralized distributed value key store (think database). It’s using Kademlia for p2p discovery (nodes to discover and communicate with each other). It incorporates end-to-end encryption (protect your data so only you can access it). Data is stored along with verifiable Merkle Tree’s (make sure your data hasn’t been tampered with).

A competitor that comes to mind is Bluzelle, their approach is a REST based service on top of their SWARM cluster. So the designs are quite different.

Fluence almost reminds me a decentralized Redis (and in memory key value store, except Fluence uses disk storage instead of in memory). This is the next evolution step, first comes processing (EVM), then comes storage (IPFS, FileCoin, StorJ), then we start building accessibility services. Technically you could just use plain disk to store your data, but databases have made us lazy and we expect to have easily optimized interfaces to work through.

Young repo, few contributors, but looks like quite a bit of substance. Fluence is a scala project (functional programming language that evolved from Java).

Disclaimer, I’ve never done production level Scala, I’ve only used it in my Spark research projects, so my skillset here is definitely lacking.

Good structure, good logging, good naming standards, great comments.

No comments from me, I just liked it.

Using GRPC for their data packet communication layer (built on top of Protobuff), I like leveraging existing infrastructure, let them handle what they do well, and you handle what you do well.

RocksDB being used as the key value storage, it’s a fast, scalable, and safe choice.

Kademlia implemented for discovery, GRPC implemented for ProtoBuff layer.

B-Tree index, and MerkleRoot already implemented as data validation and indexed lookup (for quickly finding your data).

Already doing advanced production checks (version matching)

Good error handling.

Not seeing ownership enforcement, might still be coming.

There are references to it though.

Contract implementation. Storage providers and users need to agree on a contract implementation (how much storage space, how much redundancy, how much will it cost), once the contract is closed storage can occur. Nice implementation of how participants are searched for in a specific contract name space and how the contract is sealed. I really like this approach. The onus is on the provider to join the pool, so if there aren’t enough providers that join the pool, the contract simply does not get sealed. Can then change your contract metrics and resubmit, creates a real value add for the token utilization.

So the flow would be;

Create Storage contract -> Search providers -> Providers found (enough to match your storage and redundacy rates) -> Seal contract and provide storage key -> Use key to store key/value pairs in Rock DB.

Conclusion: In all honesty I’m really surprised. They already have more substance available than 99% of ICO’s out there, while this still needs all of the integration and ease of use implementation, the core implementation is there and it looks solid. All the key features they discuss in the whitepaper (the difficult bits) have been implemented.

From here they can focus on tokenization (which is currently absent), further enhancements, and making it consumer facing. These guys are really good. I’m impressed.
