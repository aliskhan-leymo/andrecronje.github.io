Let’s play with the wallet and the contract creator.

Contract creator running, starting up their thrift server, so what is Thrift? It’s actually a really cool Apache project, it let’s you cross compile in a variety of different languages, including C++, Java, Python, PHP, Ruby, Erlang, Perl, Haskell, C#, Cocoa, JavaScript, Node.js, Smalltalk, OCaml and Delphi and other languages and it’s a transfer protocol, like ProtoBuff.

So far, I’m not seeing anything new from the Credits team, but they are using a bunch of cool technologies, this is less a ground up build and more a Frankenstein approach (I like a good Frankenstein approach though), let’s continue;

This is looking better, and this is very good news, because it means we should be able to write a script to mass send transactions in a bit, but let’s not get overexcited just yet.

We have a look at sendTransaction, it does two things, it sends the transaction you want to send, and then it sends the FEE to the fee wallet. Guys, don’t do this, I’m going to implement this file, and just remove the FEE section, fee needs to be internal to the transfer mechanism inside of nodes when applying state or UTXO, not external. This raises the question about if there is actually a consensus engine running in the background, since that would mean right now it’s just completely ignoring fee’s? But let’s continue.

sendTransactionInternal is the important one afterall, so, this is definitely better than what they had, unfortunately it’s still not particularly good. The exception handling is also rushed, but they did call this a pre-beta. I have to admit, this reads more like a PoC to me, but let’s continue.

The Thrift code is just boiler plate so we can move on.

Usercode is just saved locally as a .java file in your local directory. ContractExecutorServiceImpl.java wasn’t implemented (so that explains our earlier null pointer)

So right now all this contract does is save a java file locally.

The rest is more boilerplate, let’s look at the wallet

AppState was moved out of the main, this is a step in the right direction, but still a far removed from how it should be done. But, improvement none the less.

No. Just. No. Come on Credits team, you guys are better than this, let’s have a telegram chat or something and discuss this architecture and design, but it hurts me to keep seeing implementations like this, as a blockchain community we can do a lot better and we should do a lot better!

Ok, it’s calling transactionFlow, let’s have a look;

We are using Thrift’s TProtocol to send transactions using Thrift Service Client. Thrift is being used to obfuscate most of the actual communication leg, so difficult to see the internals.

So let’s quickly talk about Thrift throughput;

From most TPS test, Thrift does not perform as well as ProtoBuff or eProsima, so I’m not sure if this is going to give them their throughput they want, I think they would have been a lot faster with a custom asn.1 RPC format (or the standard RLP for that matter). I’m just not seeing the innovation here.

New coins are created and saved locally in a CSV.

Ok guys, I’m done for now, I really loved that the guys from Credits gave me the client early, and I really wanted to like the code more, but it’s still basic and unfortunately junior code. They did improve from last time, but it’s still so far away from being something real, we aren’t at PoC level yet, we can’t be talking about beta already.

Conclusion: They did update MD5 to Blake, but the rest is unfortunately still no where near the level it needs to be able to compete in this field or against the caliber of developers out there. Credits team, let’s talk, I want to help
