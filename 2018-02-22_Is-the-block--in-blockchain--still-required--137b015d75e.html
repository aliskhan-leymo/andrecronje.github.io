Is the block, in blockchain, still required?

I’ve been playing with this idea now for awhile. Is the block, in blockchain, still required? So let’s look at the original blockchain proposal. We see the cornerstone of this proposal is Proof of Work. Proof of Work is essentially it’s own worst enemy. So what made Proof of Work great? It means a lot of computational power is required, it deceases the chances for centralization since the chances of guessing the hash are essentially random spread out across a big enough sector. That is great for consensus, it means a lot of work to accomplish the hash and very little work to validate the hash. So the capability to cheat the system turned into the 51% attack problem/solution, dependent on how you see it.

But, this means PoW is inherently slow, at current capacity for Bitcoin around 10 minutes. So how do you increase transactional throughput if you can only confirm 1 transaction every 10 minutes? Easy, you package them all in a block. So the block, in blockchain, exists to overcome this limitation, it’s a scaling mechanism for PoW.

Currently, we are seeing more and more consensus mechanisms coming out. The current flavor of the month is definitely Proof of Stake, followed closely by mechanisms such as dBFT, or even PoA. So what’s the difference here? So in PoW you have every miner trying to compete and to win as fast as possible. In PoS you have a lottery of who gets to seal the block. If you have more stake, you can buy more lottery tickets, if you’ve been doing this longer than the rest, your chances of winning are higher, and then lastly, that bit of randomness. So now the winning node gets chosen and it’s there turn to seal the block. Why is this still a block?

So one reason why a block is still good, is it’s easier to transmit a single set of data p2p than it is to submit a bunch of smaller transactions. Transferring 1 1MB block every few minutes is definitely more preferred than transferring 1024 1KB blocks throughout a minute, right? I think wrong, we are already doing it when we are transferring the actual transactions between each node, so the overhead is essentially the same.

So what I’ve started building, is a transaction chain. I’m throwing away the blocks. This works the same as the block mechanism, but on a transaction level. A transaction still references the previous transaction hash before it and we keep the same chain mechanism. But instead, a chosen validator in a non PoW scheme can now stream transactions as quickly as it can for a set amount of time. The stream itself will be FIFO to ensure maximum throughput. But this will essentially decrease confirmations time to a negligible number. Now we have the problem of out of sync nodes, but the issue is exactly the same be it blocks or transactions.

Currently busy with the PoC, let’s see how far I get and how long it takes.
